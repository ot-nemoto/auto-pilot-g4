# コールドスタート分析 & モデル配置戦略（コスト込み）

## 前提条件（実測値に更新）

Gemma4の実際のモデルサイズ確認により、当初想定から変更。

| 項目 | 当初想定 | 実際 |
|------|---------|------|
| Gemma4モデルサイズ | ~4.3GB | **~9.6GB** |
| Dockerイメージ合計 | ~6.4GB | **~11.6GB** |

---

## 起動フェーズの分解

```
ジョブ投入
  │
  ├─[A] EC2インスタンス起動          60〜90秒  （全戦略共通）
  ├─[B] Dockerイメージ pull (ECR)    ？        ★ 戦略によって変わる
  ├─[C] コンテナ起動・初期化         5〜10秒
  ├─[D] Gemma4 モデルロード          ？        ★ 戦略によって変わる
  └─[E] Playwright起動              3〜5秒
         │
         テスト開始 〜 テスト完了（平均5分と仮定）
         │
         インスタンス終了（課金終了）
```

---

## 4戦略の比較

### 戦略1: 毎回ECR pull（モデル内包）

```
EC2起動 → ECR pull（11.6GB） → コンテナ起動 → テスト → 終了
```

| フェーズ | 時間 |
|----------|------|
| A: EC2起動 | 75秒 |
| B: ECR pull（11.6GB） | **150〜240秒** |
| C+D+E: 起動〜テスト開始 | 15秒 |
| **合計コールドスタート** | **4〜6分** |

**メリット**: CI/CDがシンプル（ECR pushだけ）  
**デメリット**: テストより待ち時間の方が長くなる可能性がある

---

### 戦略2: EFS分離（モデルをEFSに配置）※再評価

モデルが9.6GBになったことで、以前の試算から**逆転して有利**になった。

```
EC2起動 → ECR pull（2GB） → EFSマウント → モデルロード → テスト → 終了
```

| フェーズ | 時間 |
|----------|------|
| A: EC2起動 | 75秒 |
| B: ECR pull（2GB） | **25〜50秒** |
| C: EFSマウント | 5秒 |
| D: モデルロード（EFSから） | 60〜90秒 |
| E: Playwright起動 | 4秒 |
| **合計コールドスタート** | **2.8〜3.7分** |

**メリット**: イメージが軽くpullが大幅短縮、モデル更新がEFSコピーだけで完結  
**デメリット**: EFS固定費（9.6GB × $0.30 = **$2.88/月**）

---

### 戦略3: EC2 Warm Pool

EC2を**停止状態**でプールしておき、ジョブが来たら再開する。モデルはEBSに残るので再ロード不要。

```
平時: EC2停止（EBSコストのみ）
ジョブ来たら: 停止→起動（約1分）→コンテナ起動→テスト開始
```

| フェーズ | 時間 |
|----------|------|
| A: EC2再開（停止→起動） | **30〜60秒** |
| B: ECR pull → **スキップ**（Dockerキャッシュ済み） | **0秒** |
| C+D+E: 起動〜テスト開始 | 15秒 |
| **合計コールドスタート** | **約1分** |

**メリット**: 最速。モデルpull・ロード不要  
**デメリット**: EBSコスト（11.6GB相当: **$0.93/月**）、Auto Scalingの設定が必要

---

### 戦略4: EC2 常時起動（サーバレス断念）

g4dn.xlargeを停止せず常時稼働させる。

| フェーズ | 時間 |
|----------|------|
| **合計コールドスタート** | **ほぼ0秒** |

**メリット**: 即時実行  
**デメリット**: g4dn.xlarge Spot ~$0.19/h → **月約$137**（常時稼働）

---

## コスト比較（月100回・テスト5分）

### 前提
- g4dn.xlarge Spot: $0.19/時間
- 請求時間 = コールドスタート + テスト実行時間

| 戦略 | 課金時間/回 | EC2費用/月 | 固定費/月 | **合計/月** |
|------|-----------|-----------|---------|-----------|
| 1: 毎回ECR pull | 10分 | $3.17 | $1.16（ECR） | **$4.33** |
| 2: EFS分離 | 8分 | $2.53 | $1.16+$2.88 | **$6.57** |
| 3: Warm Pool | 6分 | $1.90 | $0.93（EBS） | **$2.83** |
| 4: 常時起動 | - | $137 | - | **$137** |

---

## 実用性の比較

| 戦略 | コールドスタート | 月100回コスト | CI/CD複雑度 | 実用性 |
|------|---------------|-------------|-----------|--------|
| 1: 毎回ECR pull | 4〜6分 | $4.33 | ◎ シンプル | △ テストより待ち時間が長い |
| 2: EFS分離 | 2.8〜3.7分 | $6.57 | ◎ シンプル | ○ まだ長い |
| 3: Warm Pool | 約1分 | $2.83 | ○ やや複雑 | ◎ 現実的 |
| 4: 常時起動 | ほぼ0秒 | $137 | ◎ シンプル | △ コスト高 |

---

## 結論（9.6GBモデル前提での再評価）

**戦略3（Warm Pool）を推奨。**

モデルが9.6GBになったことで、純粋なコールドスタート型（戦略1）は待ち時間が長すぎて実用的でない。
コスト・速度・シンプルさのバランスが最も良いのがWarm Pool。

```
推奨:
  AWS Batch + EC2 Warm Pool
  ・平時: EC2を停止状態でプール（EBSコスト$0.93/月のみ）
  ・実行時: 停止→起動（約1分）→テスト開始
  ・完了後: EC2を再び停止状態に戻す
```

### フェーズ別移行計画

```
Phase 1（PoC・動作確認）:
  戦略1: 毎回ECR pull
  目的: インフラをシンプルに保ち、まず動くものを作る
  許容: コールドスタートが長くても動作確認フェーズでは問題なし

Phase 2（運用開始）:
  戦略3: Warm Pool に移行
  目的: 実用的な待ち時間（1分）を実現
```

## 参考: EFS再評価

当初（モデル4.3GB想定）はコスト的に不利と判断したが、9.6GBでは試算が変わる。

| モデルサイズ | EFS固定費 | ECR pull短縮効果 | 判定 |
|------------|---------|----------------|------|
| 4.3GB（当初） | $1.29/月 | EC2節約$0.03/月 | ❌ 不利 |
| 9.6GB（実際） | $2.88/月 | EC2節約$0.64/月 | ❌ まだ不利 |

EFSはWarm Poolと比べても速度・コスト両面で劣るため採用しない。
