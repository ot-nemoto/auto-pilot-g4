# コールドスタート分析 & モデル配置戦略（コスト込み）

## 起動フェーズの分解

```
ジョブ投入
  │
  ├─[A] EC2インスタンス起動          60〜90秒  （全戦略共通）
  ├─[B] Dockerイメージ pull (ECR)    ？        ★ 戦略によって変わる
  ├─[C] コンテナ起動・初期化         5〜10秒
  ├─[D] Gemma4 モデルロード          ？        ★ 戦略によって変わる
  └─[E] Playwright起動              3〜5秒
         │
         テスト開始 〜 テスト完了（平均5分と仮定）
         │
         インスタンス終了（課金終了）
```

---

## 3戦略のコールドスタート時間

### イメージサイズの違い

| 戦略 | イメージ内容 | ECRイメージサイズ |
|------|-------------|-----------------|
| 1: 毎回ECR pull | モデル込み | ~6.4GB |
| 2: AMIキャッシュ | モデル込み（ただしpullスキップ） | ~6.4GB（不使用） |
| 3: EFS分離 | モデルなし | ~2GB |

### フェーズ別時間（g4dn.xlarge想定）

| フェーズ | 戦略1 | 戦略2（AMI） | 戦略3（EFS） |
|----------|-------|-------------|------------|
| A: EC2起動 | 75秒 | 75秒 | 75秒 |
| B: ECR pull | 90秒 | **0秒**（スキップ） | 40秒 |
| C: コンテナ起動 | 7秒 | 7秒 | 7秒 |
| D: モデルロード | 3秒（NVMe） | 3秒（NVMe） | 45秒（EFS） |
| E: Playwright起動 | 4秒 | 4秒 | 4秒 |
| **合計コールドスタート** | **179秒（約3分）** | **89秒（約1.5分）** | **171秒（約2.9分）** |

## コスト計算（月100回実行）

### 前提

- g4dn.xlarge Spot: $0.19/時間（= $0.00317/分）
- テスト実行時間: 5分（固定）
- 請求時間 = コールドスタート + テスト実行時間

### 1回あたりのEC2課金時間と費用

| 戦略 | 課金時間 | 1回のEC2費用 | 月100回EC2費用 |
|------|----------|-------------|--------------|
| 1: 毎回pull | 3.0 + 5 = **8.0分** | $0.025 | **$2.53** |
| 2: AMIキャッシュ | 1.5 + 5 = **6.5分** | $0.021 | **$2.06** |
| 3: EFS分離 | 2.9 + 5 = **7.9分** | $0.025 | **$2.50** |

### インフラ固定費（月額）

| 戦略 | 追加インフラ | 月額固定費 |
|------|------------|----------|
| 1: 毎回pull | なし | **$0** |
| 2: AMIキャッシュ | EBSスナップショット 6.4GB | **$0.32** |
| 3: EFS分離 | EFS 4.3GB | **$1.29** |

### 月100回のトータルコスト

| 戦略 | EC2費用 | 固定費 | **合計** |
|------|---------|-------|---------|
| 1: 毎回pull | $2.53 | $0 | **$2.53** |
| 2: AMIキャッシュ | $2.06 | $0.32 | **$2.38** |
| 3: EFS分離 | $2.50 | $1.29 | **$3.79** |

→ **EFS分離は一番コスト高。AMIキャッシュが最安だが差は$0.15。**

## 実行頻度別 推奨戦略

```
月100回以下:
  → 戦略1（毎回ECR pull）
     差額が$0.15〜$1.26 で誤差範囲。CI/CDをシンプルに保つ方が価値高い。

月1,000回前後:
  → 戦略2（AMI）が$4安くなるが、CI/CD複雑化のコストと相談。
    実行頻度が読めてきたら移行を検討。

月10,000回以上:
  → 戦略2（AMI）が$47安い。CI/CD自動化の工数を回収できる。
```

---

## EFS分離はいつ有効か

コスト面では不利だが、以下の場合は有利になる：

- **複数の異なるモデルを使い分けたい**（EFSに複数モデルを置いてスイッチ）
- **モデル更新頻度が高い**（EFSへのコピーだけで更新完了、AMI再作成不要）
- **Spot中断後の再実行コストを下げたい**（イメージが軽い分pullが速い）

## 結論

| 観点 | 推奨 |
|------|------|
| **現時点（PoC・月100回以下）** | **戦略1: 毎回ECR pull** |
| **運用安定後（月1,000回超）** | **戦略2: AMIキャッシュ**（移行検討） |
| **EFS** | コスト的に不利のため採用しない |

### 理由

- 月100回では3戦略の差は最大**$1.26/月**で誤差範囲
- EFSはモデルロードが遅い（NVMe比較で42秒遅い）うえに月$1.29の固定費で二重に不利
- 最初はシンプルな戦略1で始め、実行頻度が増えたらAMI自動化を検討するのが合理的

### フェーズ別移行計画

```
Phase 1（PoC）:
  戦略1: 毎回ECR pull
  CI/CD: git push → docker build → ECR push → 完了

Phase 2（実行頻度 月1,000回超になったら）:
  戦略2: AMI自動作成をGitHub Actionsに追加
  CI/CD: git push → docker build → ECR push
              → EC2起動 → docker pull → AMI作成
              → Batch Compute Environment 更新
```

## 参考: Ondemandの場合

Spot中断が許容できない場合（テスト失敗が困る本番CI連携等）はOndemand。
コスト差は大きくなるため、戦略2（AMI）の優位性が上がる。

| 戦略 | 月100回（Ondemand） | 月1,000回（Ondemand） |
|------|-------------------|---------------------|
| 1: 毎回pull | $7.0 | $70 |
| 2: AMI | $5.7 | $57 |
| 3: EFS | $7.3 | $73 |
